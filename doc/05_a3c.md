# A3C (Asynchronous Advantage Actor-Critic)

## Core idea
A3C extends actor-critic by running multiple workers in parallel, each interacting with its own environment stream. Asynchrony decorrelates experience and speeds wall-clock learning.

Unlike single-worker on-policy training, A3C trades perfectly fresh gradients for higher throughput and broader state visitation.

## How this repo implements it
- Module: `benchmarks/a3c.py`
- Runner: `a3c_benchmark.py`
- Pattern: worker processes collect short rollouts and push batches via queue to a central learner.
- Global model parameters are periodically shared back to workers.

## Training flow
1. Spawn `N` workers.
2. Each worker samples rollout chunks (`rollout_steps`).
3. Learner consumes batches, computes actor-critic loss, updates global net.
4. Workers refresh local params from global state.
5. Episode rewards are aggregated centrally.

This introduces a controlled amount of policy lag: workers may act with slightly stale parameters while the learner is updating.

## Strengths
- Better decorrelation than single-process on-policy methods.
- Good CPU parallelism.
- Historically important precursor to many modern parallel RL systems.

## Limitations
- More complex engineering (queues/process lifecycle).
- Potential instability from stale gradients/parameter lag.
- Harder to reproduce exactly due to async timing.

## Common tuning levers
- **Worker count**: increase until CPU utilization is high but not saturated.
- **Rollout length**: shorter rollouts reduce lag; longer rollouts reduce IPC overhead.
- **Learning rate**: async noise can require lower rates than A2C.

## Practical notes
- Keep worker count proportional to CPU cores.
- Too many workers can increase update lag.
- Use shorter rollouts for faster parameter refresh.
