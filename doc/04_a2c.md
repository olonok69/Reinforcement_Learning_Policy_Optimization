# A2C (Advantage Actor-Critic)

## 1) Why A2C exists
A2C is designed to fix a key weakness of vanilla policy gradient (REINFORCE): **high gradient variance**.

In REINFORCE, a common estimator is:

$$
\nabla_\theta J(\theta) \propto \mathbb{E}\left[\nabla_\theta \log \pi_\theta(a_t|s_t)\,G_t\right]
$$

with full-trajectory return:

$$
G_t = r_{t+1} + \gamma r_{t+2} + \gamma^2 r_{t+3} + \dots
$$

This estimator is unbiased, but noisy. Two episodes that start from similar states can produce very different returns, so policy updates can oscillate and learning can be sample-inefficient.

---

## 2) Actor-Critic idea
Actor-Critic methods combine:

- **Actor**: policy $\pi_\theta(a|s)$ (chooses actions)
- **Critic**: value estimator (evaluates outcomes)

The critic gives a baseline so the actor learns from **relative quality** of actions, not only raw returns.

### Advantage signal
A2C centers learning around the advantage function:

$$
A^\pi(s_t,a_t)=Q^\pi(s_t,a_t)-V^\pi(s_t)
$$

Interpretation:
- If $A(s,a) > 0$, the action was better than expected for that state â†’ increase its probability.
- If $A(s,a) < 0$, the action was worse than expected â†’ decrease its probability.

This is the core variance-reduction intuition behind A2C.

---

## 3) Practical advantage estimation
Directly computing both $Q(s,a)$ and $V(s)$ can be expensive, so in practice we estimate advantage.

A common one-step estimator uses TD error:

$$
y_t = r_t + \gamma V_\phi(s_{t+1})
$$

$$
\delta_t = y_t - V_\phi(s_t)
$$

$$
\hat{A}_t \approx \delta_t
$$

More generally, implementations may use n-step returns or GAE-style estimators.

---

## 4) A2C objective
A2C usually optimizes three terms:

### Policy loss

$$
\mathcal{L}_{policy} = -\mathbb{E}\left[\log\pi_\theta(a_t|s_t)\,\hat{A}_t\right]
$$

### Value loss

$$
\mathcal{L}_{value} = \mathbb{E}\left[(V_\phi(s_t)-\hat{R}_t)^2\right]
$$

### Entropy bonus

$$
\mathcal{L}_{entropy} = \mathbb{E}\left[\mathcal{H}(\pi_\theta(\cdot|s_t))\right]
$$

Total loss (sign convention may vary by implementation):

$$
\mathcal{L} = \mathcal{L}_{policy} + c_v\mathcal{L}_{value} - c_e\mathcal{L}_{entropy}
$$

where:
- $c_v$ controls critic fitting strength
- $c_e$ controls exploration pressure

---

## 5) A2C vs A3C
Both come from the same actor-critic family.

- **A3C**: asynchronous workers update with possible parameter lag.
- **A2C**: synchronous update step (typically over a batch from parallel environments).

A2C is often easier to reason about and usually more reproducible, while still benefiting from actor-critic variance reduction.

---

## 6) Canonical training loop (conceptual)
1. Collect rollout data using current policy.
2. Compute targets and advantage estimates.
3. Update actor and critic jointly with combined loss.
4. Repeat with updated policy.

In vectorized setups, data comes from multiple environments synchronously before each update.

## 7) How this repository implements A2C
- Module: `benchmarks/a2c.py`
- Runner: `a2c_benchmark.py`
- Environment: `CartPole-v1`

### Architecture
- One shared trunk network
- Two heads:
  - policy logits head (actor)
  - scalar value head (critic)

### Update style used here
This repo uses an **A2C-style episodic actor-critic** variant:
- Collect one full episode.
- Compute Monte Carlo discounted returns.
- Estimate advantages as:

$$
\hat{A}_t = G_t - V(s_t)
$$

- Optimize policy loss + value loss - entropy bonus.

So, conceptually it follows A2C (advantage actor-critic with shared actor/critic network), while using per-episode returns instead of a multi-env synchronous rollout collector.

---

## 8) Strengths and limitations
### Strengths
- Lower variance than REINFORCE.
- Better training stability from critic baseline.
- Simple and effective bridge to PPO/TRPO-style methods.

### Limitations
- Still on-policy (can be less sample-efficient than replay-based methods).
- Sensitive to coefficient tuning (`value_coef`, `entropy_coef`) and learning rate.
- Critic quality strongly affects actor update quality.

---

## 9) Practical tuning checklist
- If policy collapses early: increase entropy coefficient slightly.
- If learning is noisy: lower learning rate and verify value loss scale.
- If critic underfits: increase critic capacity or adjust value coefficient.
- If exploration is too random for too long: reduce entropy coefficient gradually.
- Always compare across multiple seeds before concluding performance differences.
